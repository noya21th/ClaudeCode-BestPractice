## Архитектурные антипаттерны

### 1. Монолитный CLAUDE.md

**Проблема**: Единственный файл CLAUDE.md длиной более 500 строк, в котором смешаны инструкции по установке, соглашения по написанию кода, документация API, шаги деплоя и советы по отладке.

**Симптомы**: Claude игнорирует инструкции ближе к концу файла. Непоследовательное соблюдение соглашений. Раздувание контекста при старте сессии.

**Решение**: Разбейте файл на сфокусированные части. Оставьте в корневом CLAUDE.md не более 200 строк с общими соглашениями. Используйте компонентные CLAUDE.md-файлы для деталей. Применяйте `.claude/rules/*.md` с glob-паттернами для правил, привязанных к типам файлов.

```
# До (монолитный)
CLAUDE.md              # 800 строк, покрывающих всё

# После (разделённый)
CLAUDE.md              # 150 строк — общие соглашения, обзор проекта
frontend/CLAUDE.md     # 80 строк — паттерны React, структура компонентов
backend/CLAUDE.md      # 80 строк — соглашения по API, паттерны работы с БД
.claude/rules/ts.md    # Правила TypeScript (Glob: **/*.ts)
.claude/rules/test.md  # Соглашения по тестированию (Glob: **/*.test.*)
```

### 2. Agent для всего подряд

**Проблема**: Использование Agent для простых задач, которым достаточно инструкций, вставленных в текущий контекст.

**Симптомы**: Ненужная изоляция контекста. Результаты не видны в основном диалоге. Медленное выполнение из-за накладных расходов на запуск Agent. Трата токенов на дублирование контекста.

**Решение**: Используйте Skill для переиспользуемых знаний и процедур. Используйте Command для шаблонов промптов, вызываемых пользователем. Оставьте Agent для задач, которым действительно нужна изоляция, память, ограничение инструментов или фоновое выполнение.

### 3. Отсутствие изоляции контекста

**Проблема**: Выполнение длительных исследовательских задач (исследование, код-ревью, анализ данных) прямо в основном диалоге без изоляции.

**Симптомы**: Контекст диалога быстро переполняется. Нерелевантные промежуточные результаты загрязняют будущие ответы. Приходится часто использовать `/compact`.

**Решение**: Используйте Agent для задач, генерирующих большой объём промежуточных результатов. Agent работает в чистом контексте и возвращает только итоговую сводку.

### 4. Циклический запуск Agent

**Проблема**: Agent A порождает Agent B, который порождает Agent A, создавая бесконечный цикл.

**Симптомы**: Достигнут лимит `maxTurns`. Исчерпан бюджет токенов. Полезного результата нет.

**Решение**: Проектируйте DAG-воркфлоу (направленный ациклический граф). Используйте поле `tools:` для ограничения того, каких Agent может порождать подагент. Проверяйте определения Agent на циклические ссылки `Agent(...)`.

```yaml
# Ограничение порождаемых Agent
---
name: planner
tools: Read, Glob, Grep, Agent(implementer)
# planner может породить только implementer, но не самого себя
---
```

### 5. Хардкод имён моделей

**Проблема**: Использование полных идентификаторов моделей вроде `claude-sonnet-4-20250514` в определениях Agent.

**Симптомы**: Agent ломаются при устаревании идентификаторов моделей. Непоследовательное поведение в разных окружениях.

**Решение**: Используйте алиасы: `haiku`, `sonnet`, `opus` или `inherit`. Они всегда указывают на последнюю версию.

## Антипаттерны конфигурации

### 6. Слишком широкие разрешения

**Проблема**: Установка `"permissionMode": "bypassPermissions"` на всех Agent и использование `--dangerously-skip-permissions` по умолчанию.

**Симптомы**: Непроверенные удаления файлов. Непреднамеренный `git push --force`. Риски безопасности из-за непроверенных MCP-инструментов. Отсутствие журнала одобрений.

**Решение**: Используйте `acceptEdits` для Agent, которым нужно записывать файлы, но не выполнять произвольные команды. Используйте `plan` для Agent, занимающихся только исследованием. Оставьте `bypassPermissions` исключительно для CI/CD-окружений с песочницей.

| Режим | Правки | Bash | Лучше всего для |
|------|-------|------|----------|
| `default` | Запрос | Запрос | Интерактивная разработка |
| `acceptEdits` | Авто | Запрос | Agent, пишущие файлы |
| `plan` | Запрет | Запрет | Agent для исследования и планирования |
| `bypassPermissions` | Авто | Авто | Только CI/CD с песочницей |

### 7. Игнорирование иерархии настроек

**Проблема**: Определение настроек в неправильном файле с последующим удивлением, почему они переопределены или игнорируются.

**Симптомы**: Hook не срабатывают. Разрешения не соответствуют ожиданиям. У членов команды разное поведение.

**Решение**: Разберитесь в порядке приоритета (побеждает высший):

1. **Управляемые** (`managed-settings.json`) — принудительные от организации
2. **Флаги CLI** — переопределение на одну сессию
3. **Локальные** (`.claude/settings.local.json`) — личные настройки проекта
4. **Общие** (`.claude/settings.json`) — командные настройки проекта
5. **Глобальные** (`~/.claude/settings.json`) — личные значения по умолчанию

### 8. Все Hook синхронные

**Проблема**: Каждый Hook выполняется синхронно, блокируя ответ Claude во время работы скриптов.

**Симптомы**: Медленные ответы. Заметная пауза после каждого вызова инструмента. Пользователь воспринимает Claude как тормозящий.

**Решение**: Установите `"async": true` на Hook, выполняющих побочные эффекты (логирование, уведомления, телеметрия). Оставляйте синхронными только те Hook, которым нужно контролировать или изменять поведение (проверки безопасности, решения по разрешениям).

### 9. Отсутствие settings.local.json

**Проблема**: Личные предпочтения (звуковые уведомления, пользовательские пути, логирование отладки) коммитятся в `.claude/settings.json`, где они влияют на всю команду.

**Симптомы**: Коллеги слышат ваши звуки уведомлений. Hook для отладки запускаются в продакшене. Персональные MCP-серверы видны всем.

**Решение**: Используйте `.claude/settings.local.json` для личных переопределений. Убедитесь, что файл добавлен в `.gitignore`. Делитесь через общие настройки только теми Hook и параметрами, которые нужны всей команде.

## Антипаттерны Skill

### 10. user-invocable: true для Skill, предназначенных только для Agent

**Проблема**: Skill, предназначенный только для предзагрузки в Agent (через поле `skills:`), отображается в пользовательском меню `/` (слэш-команды).

**Симптомы**: Пользователи вызывают Skill напрямую и получают запутанные результаты, потому что он задумывался как фоновые знания для Agent.

**Решение**: Установите `user-invocable: false` для Skill, предназначенных только для предзагрузки в Agent.

```yaml
---
name: internal-api-conventions
user-invocable: false
description: "REST API conventions for the internal platform"
---
```

### 11. Отсутствие описания

**Проблема**: У Skill нет поля `description` во frontmatter.

**Симптомы**: Claude не может автоматически обнаружить Skill. Он срабатывает только когда пользователь явно вводит `/slash-command`. Skill невидим для логики автоматического вызова.

**Решение**: Всегда добавляйте `description`, объясняющий **когда** использовать Skill. Используйте ключевые слова, с которыми Claude столкнётся естественным образом: "Use when creating React components" или "Triggers on Python test files."

### 12. Гигантские файлы Skill

**Проблема**: Единственный файл SKILL.md длиной более 1000 строк, покрывающий все крайние случаи, примеры и справочные материалы.

**Симптомы**: Раздувание контекста при загрузке Skill. Claude может игнорировать инструкции ближе к концу. Медленная загрузка при предзагрузке в Agent.

**Решение**: Сосредоточьте SKILL.md на основной процедуре (до 200 строк). Вынесите справочные материалы в файлы `references/` внутри директории Skill. Используйте импорт `@path` для дополнительного контента, который Claude загружает по требованию.

### 13. disable-model-invocation на часто нужных Skill

**Проблема**: Установка `disable-model-invocation: true` на Skill, который Claude должен обнаруживать и вызывать автоматически.

**Симптомы**: Claude никогда не вызывает Skill автоматически. Пользователям приходится всегда вводить `/slash-command` вручную, даже когда контекст явно требует этот Skill.

**Решение**: Используйте `disable-model-invocation: true` только для Skill, которые вызываются исключительно пользователем (опасные операции, деплой и т.д.). Для Skill, которые Claude должен обнаруживать самостоятельно, оставьте параметр не установленным или задайте `false`.

## Операционные антипаттерны

### 14. Никогда не сжимать контекст

**Проблема**: Работа в длительной сессии без сжатия, при которой контекст заполняется до 100%.

**Симптомы**: Claude начинает забывать предыдущие инструкции. Ответы становятся менее точными. Сессия в конечном итоге завершается ошибкой переполнения контекста.

**Решение**: Запускайте `/compact` вручную при примерно 50% использования контекста. Используйте `/context` для визуализации текущего использования. Для длительных задач разбивайте работу на подзадачи, каждая из которых завершается в пределах 50% ёмкости контекста.

### 15. Отсутствие циклов проверки

**Проблема**: Принятие результатов Claude без проверки работоспособности — без запуска тестов, проверки типов, визуального осмотра.

**Симптомы**: Сгенерированный код содержит синтаксические ошибки. API-интеграции используют неправильные эндпоинты. Конфигурационные файлы содержат опечатки.

**Решение**: Всегда проверяйте результат. Запускайте тесты после генерации кода. Используйте автоматизацию браузера (Chrome, Playwright) для визуальной проверки. Включайте шаги проверки в ваши Command и определения Agent.

### 16. Игнорирование лимитов запросов

**Проблема**: Планирование крупных мультиагентных задач без учёта бюджета токенов и лимитов запросов.

**Симптомы**: Agent достигают лимитов посреди задачи. Работа теряется или остаётся незавершённой. Неожиданные расходы.

**Решение**: Используйте `/usage` для проверки текущего статуса лимитов. Планируйте задачи так, чтобы они укладывались в бюджет. Используйте `model: haiku` для высоконагруженных подзадач низкой сложности. Разбивайте крупные задачи на меньшие сессии.
